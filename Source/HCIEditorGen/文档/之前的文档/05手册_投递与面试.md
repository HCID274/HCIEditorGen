# UE实践手册：投递与面试（围绕 HCIEditorGen / 工具向）

目标：让作品“可被快速理解、可被深挖、可被信任”，并且把每一次踩坑都变成面试素材。

---

## 1. 作品包装（只讲一个主项目）

原则：
1. 简历只讲一个主项目：`HCIEditorGen`（并在同一项目下用 `HCIDuel` 做 dogfooding 验收底座；不要拆成第二个完全独立的叙事）。
2. 亮点必须能被证明：截图/视频/数据/可复现步骤缺一不可。
3. 工具岗更看重：**规模化（量产）+ 稳定性 + 可用性 + 落地推广**，而不是玩法堆砌。

最小交付包：
1. 90 秒视频（工具为主，Runtime 为辅）：
   - SourceData（AbilityKit）改动（例如 `params.damage`）→ Import/Reimport → Derived 资产更新（必须出现右键 `Reimport`）
   - **失败用例 1 个**（脏数据/缺字段/断网/超时）→ 清晰报错 + fallback（不脏写）
   - 批处理演示（至少 50+ 条）→ 进度/统计/失败清单
   - 最后 10 秒：`HCIDuel` 抽样验证（工具产物在 Runtime 生效）
2. `README.md`：如何运行、核心模块、复现方式（含失败用例/批处理）、约束与已知问题
3. 证据链素材包：
   - Reimport 录屏/截图（前后对比）
   - 批处理报告（统计 + 失败清单）
   - 错误用例与日志（能让面试官看到“你怎么保护编辑器稳定”）
4. STAR 故事：**效率/稳定性/取舍** 各 1 条（必须能对照素材包）

### 1.1 简历子弹点（示例，按你实际实现改）
Tools/Pipeline（主项目）：
1. UE 编辑器工具：开发 `HCIEditorGen` 插件/Editor Module，建立 **Schema-first** 的 SourceData→Import/Reimport→Derived 资产管线；支持批量导入、严格校验、差异预览（Dry-run）与失败清单/统计报告，实现“量产可控、失败不脏写”。
2. 稳定性工程：为断网/超时/脏数据设计 fallback（Mock Provider）与错误聚合输出，保证编辑器不崩、不假写，支持可复现的失败用例与定位日志。
3. （可选加分）资源检查：命名/目录/引用合法性/循环依赖检测，输出可跳转定位的检查报告。

Runtime（dogfooding 验收底座）：
1. UE5.4 + C++：构建可玩战斗样机 `HCIDuel`，用 GAS 最小闭环消费工具产物，实现“工具产物在 Runtime 生效”的抽样验收与回归验证入口。
   - 注：`HCIDuel` 推荐基于 UE 官方示例 **Lyra Starter Game** 做最小改造（节省时间、降低风险），你的原创价值集中在“验收入口 + 工具产物的运行时验证口径”。

## 2. STAR 模板（直接照填）

每条控制在 8-12 行，能 1 分钟讲完。

S（情境）：当时遇到什么问题（掉帧/行为抖动/漏判/崩溃）？  
T（任务）：你需要达成什么指标（稳定复现、帧耗下降、判定可靠）？  
A（行动）：你怎么定位（调试面板/调用栈/日志/Insights），怎么改（降频/状态复位/去重）？  
R（结果）：数据或现象怎么变化（前后对比），学到什么可迁移的方法？

## 3. 简历条目写法（把“功能”升维成“工程”）

只写 2-3 条，但每条都能被追问 5 分钟：
1. **架构层（AI 工程化）**：C++ 定制 BT 节点 + C++ 自定义 EQS（Context/Generator）实现战术选点，并接入战斗决策链路（可复现）。
2. **性能层（证据链）**：用 Unreal Insights 定位到具体函数/作用域的 CPU Cost，制造可控 Spike 并完成 Before/After 修复对比，输出报告与截图。
3. **工程层（可维护/可验证）**：模块化拆分战斗/AI/动画命中窗口，提供可运行验证步骤与素材包（README/视频/截图/报告）。

## 4. 面试常见深挖点（你要提前准备好答案）

工具/管线：
1. 你为什么坚持 Schema-first？Source of Truth 如何落地？如何处理版本演进/兼容？
2. Import/Reimport 链路怎么实现？为什么右键 Reimport 能稳定工作？Source 路径/元数据怎么存？
3. 失败不脏写怎么做？（临时文件替换/回滚/事务/保存策略）
4. 批处理规模上来后怎么控风险？（Dry-run/差异预览/白名单/可重试/失败清单）
5. 你如何保证编辑器不长时间卡死？（分帧/异步/进度/可取消/主线程边界）
6. 工具的“可用性/人机功效”你做了什么？（错误聚合、可定位、默认值、防呆）
7. 你如何落地推广？（文档/培训/反馈闭环/迭代节奏）

Runtime（dogfooding）：
1. 为什么需要 HCIDuel？工具产物怎么进入 Runtime 并验证？验证口径是什么？
2. GAS 你用了哪些最小概念？（ASC/GE/Tags/Cooldown/Cost），为什么选 GAS 作为底座？
