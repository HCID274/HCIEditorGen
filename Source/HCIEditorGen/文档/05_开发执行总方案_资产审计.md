# HCIAbilityKit 开发执行总方案（资产审计主线）

> 状态：冻结（V5）  
> 更新时间：2026-02-24  
> 适用范围：`HCIAbilityKit`

## 1. 目标与边界

- 主交付：`Smart Asset Auditor`（扫描、规则审计、报告、可追踪审阅）。
- 一期原则：先把资产审计主链路做深，再做 Agent 安全编排；禁止无确认自动改写。
- 当前执行模式：以 `simulate_dry_run` 为主，真实执行不提前承诺。

## 2. 固定执行顺序

1. 先文档冻结（目标/范围/门禁）。
2. 再实现与编译。
3. 输出 UE 手测步骤与 Pass/Fail 口径。
4. 用户手测 `Pass` 后进入下一切片。

## 3. 阶段计划与状态

### Stage A-D（资产审计主链路）

- 状态：`已完成`
- A：路线与契约冻结。
- B：种子资产链路、异步扫描、中断重试、状态跳过、预览体。
- C：规则框架与首批规则、统一结果结构、JSON 报告导出。
- D：深度检查、GC 节流、吞吐与峰值内存监控。

### Stage E（Agent 安全执行）

- 状态：`已完成`
- E1~E8 已落地：
  - Tool Registry（7 工具白名单）
  - Dry-Run Diff + 定位策略
  - Confirm Gate
  - Blast Radius（`MAX_ASSET_MODIFY_LIMIT=50`）
  - All-or-Nothing
  - SourceControl Fail-Fast（含离线本地模式）
  - Mock RBAC（默认 `Guest(read_only)`）
  - LOD Safety（Nanite/类型拦截）

### Stage F（指令解析与执行编排）

- 状态：`已收官（F1~F15）`
- 已完成闭环：`Plan -> Validate -> Execute(simulate_dry_run) -> Review -> Apply/Confirm/Ticket -> SimArchive/Handoff`。

### Stage G（执行链路后续）

## 7. Stage G-SliceG10（已完成）

### 7.1 目标边界（冻结）

- `G10` 定位：Stage G 收口切片，不做真实资产写入。
- 执行模式：继续固定 `simulate_dry_run`，产出“可进入 H1 的执行就绪报告”。
- 不在本片做：
  - 不启用真实 `Tool->Execute()` 写入；
  - 不引入审批流 UI；
  - 不修改既有 Stage E/F 门禁语义。

### 7.2 输入/输出契约（冻结）

- 输入：最新 `StageGExecuteArchiveBundle`（来自 `G9`）+ `user_confirmed`。
- 输出：`StageGExecutionReadinessReport`（`G10` 新增契约，供 `H1` 作为入口前置校验）。
- `StageGExecutionReadinessReport` 最小字段：
  - `request_id`
  - `stage_g_execute_archive_bundle_id`
  - `selection_digest`
  - `stage_g_execution_readiness_digest`
  - `stage_g_execution_readiness_status`（`ready|blocked`）
  - `ready_for_h1_planner_integration`（`true|false`）
  - `execution_mode`（固定 `simulate_dry_run`）
  - `error_code/reason`
  - `summary/items`（保留 `blocked/skip_reason/object_type/locate_strategy/evidence_key/actor_path`）

### 7.3 错误码与收敛口径（冻结）

- 成功：
  - `stage_g_execution_readiness_status=ready`
  - `ready_for_h1_planner_integration=true`
  - `error_code=-`
  - `reason=stage_g_execution_readiness_ready`
- 失败：
  - `E4218 / stage_g_execute_archive_bundle_not_ready`
  - `E4219 / stage_g_execution_mode_write_not_allowed_in_g10`
  - `E4202 / *_mismatch`
  - `E4005 / user_not_confirmed`

### 7.4 安全门禁生效位置（冻结）

- Confirm：保留，`user_confirmed=false` 直接阻断（`E4005`）。
- Blast Radius：`G10` 为只读就绪校验阶段，不触发资产修改阈值判定。
- RBAC：按只读工具口径放行；未命中用户保持 `Guest(read_only)`。
- SourceControl：`G10` 不做 checkout；保持只读路径。
- LOD Safety：`G10` 不直接调用 LOD 写工具，不新增触发点。
- Transaction：`G10` 只输出就绪报告，不开启真实事务。

### 7.5 UE 手测门禁（已通过）

- 新增命令（G10 实现目标）：
  - `HCIAbilityKit.AgentExecutePlanReviewPrepareStageGExecutionReadiness [user_confirmed=0|1] [tamper=none|digest|archive|mode]`
  - `HCIAbilityKit.AgentExecutePlanReviewPrepareStageGExecutionReadinessJson [user_confirmed=0|1] [tamper=none|digest|archive|mode]`
- 必测场景：
  1. `none + user_confirmed=1`：命中 `ready=true` 与 `reason=stage_g_execution_readiness_ready`。
  2. `user_confirmed=0`：命中 `E4005/user_not_confirmed`。
  3. `tamper=digest`：命中 `E4202/selection_digest_mismatch`。
  4. `tamper=archive`：命中 `E4218/stage_g_execute_archive_bundle_not_ready`。
  5. `tamper=mode`：命中 `E4219/stage_g_execution_mode_write_not_allowed_in_g10`。
- 通过标准：
  - 上述 5 条场景均无崩溃；
  - 摘要行包含 `request_id/error_code/reason/ready_for_h1_planner_integration/validation`；
  - JSON 含全部冻结字段。
- 用户结果：`Pass`。

## 9. G10 后续推进规定（强制，防止随意推进）

- 不是“想到哪做哪”，必须按以下门禁推进：
  1. `G10-DocFreeze`：先冻结 `目标边界/契约字段/错误码/UE 手测口径`，未冻结禁止实现。
  2. `G10-Impl`：按冻结稿实现并本地编译通过，产出最小回归命令集。
  3. `G10-UEGate`：用户 UE 手测给出 `Pass/Fail`，未 Pass 禁止进入下一切片。
  4. `H1-Planning`：仅在 `G10 Pass` 后创建 `H1`，并先更新 `00 + 05` 的目标/门禁。
  5. `H1+`：每个后续切片继续执行“文档冻结 -> 实现 -> UE 手测 -> 记录”固定模板。
- `H1` 及后续切片命名与目标，不允许口头推进，必须先在 `00 + 05` 留存一条“当前切片/下一切片”定义再开始代码。

## 10. 加速执行节奏（即日起生效）

- 目标：在不降低安全门禁的前提下，把推进从“碎片化慢推进”改为“短周期强收敛”。
- 核心原则：
  1. 小步但不碎步：一个切片至少交付“可演示闭环”，不再做过细桥接切片拆分。
  2. 并行准备：当前切片开发时，同步准备下一切片文档与测试样例草稿。
  3. 固定节奏：每个切片默认按 `文档冻结(0.5d) -> 实现(1d) -> UE门禁(0.5d)` 的 2 天窗口推进。
  4. 不牺牲门禁：`Pass` 规则不变，未 Pass 不跨切片。

### 10.1 快速主线（冻结）

- `G10`（当前）：执行链路收口切片（先文档冻结，再实现）。
  - 交付：`G10` 契约、错误码、UE 门禁清单、实现与回归。
  - 退出条件：用户 UE 手测 `Pass`。

- `H1`（紧随 G10）：LLM Planner 接入（保持 `simulate_dry_run`）。
  - 范围：只替换/扩展 `NL -> Plan JSON` 生成层，不改执行门禁链。
  - 约束：所有 LLM 输出必须继续经过 `PlanValidator + ExecutionGate`。
  - 退出条件：同一输入下可稳定生成合法 Plan；异常回退路径可用。

- `H2`（紧随 H1）：LLM 稳定性与降级策略。
  - 范围：超时、空响应、结构不合法、置信度不足时回退关键词规划器。
  - 约束：任何失败不得绕过门禁，不得触发隐式写入。
  - 退出条件：`成功/回退/失败` 三类路径均可观测、可复现。

- `I1`（紧随 H2）：最小 UI 面板（读计划/看 Diff/定位）。
  - 范围：只读为主（Plan Preview + DryRunDiff + Locate），不做审批流 UI。
  - 约束：定位策略复用现有 `camera_focus/sync_browser`。
  - 退出条件：用户可在面板完成“看计划 -> 看差异 -> 定位对象”闭环。

### 10.2 加速但受控的执行规则

- 禁止再把同一链路拆成纯“字段桥接型”微切片，除非涉及新的安全门禁。
- 每个切片开始前必须写出三行：
  1. 本片只解决什么问题；
  2. 本片不解决什么；
  3. 本片 `Pass` 的唯一判定日志。
- 每个切片结束后必须给出下一片草案（标题 + 目标 + 门禁），避免停顿等待。
