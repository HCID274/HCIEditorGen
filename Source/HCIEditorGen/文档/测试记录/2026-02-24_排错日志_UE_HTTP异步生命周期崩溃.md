# 2026-02-24 排错日志：UE HTTP 异步生命周期崩溃（黄金案例）

## 1. 现象

- 崩溃类型：`EXCEPTION_ACCESS_VIOLATION`。
- 调用栈关键点：`HCI_SendLlmHttpRequestBlocking` -> `FString::Reset()`。
- 典型地址：接近空指针（如 `0x0000000000007394`）。

## 2. 复现条件

- 在 `IHttpRequest` 回调中，使用 Lambda 引用捕获（`[&]`）函数局部变量（如 `FString& OutResponse`、`bool& bDone`）。
- 同时在函数内做“阻塞式等待”（`while(!bDone){Sleep/Tick}`）把异步请求强行改成同步。

## 3. 根因（可面试直讲）

1. `IHttpRequest` 本质异步，回调触发时机晚于函数作用域。
2. 回调里写入被引用捕获的局部变量，若该变量生命周期已结束或状态失效，会形成悬垂引用（野指针写）。
3. 阻塞轮询会引入主线程重入、Tick 竞态、卡死风险，进一步放大生命周期问题。

结论：这是典型的“异步回调写入失效栈变量”导致的内存访问违规，不是简单网络超时问题。

## 4. 反模式（Anti-Pattern）

```cpp
Request->OnProcessRequestComplete().BindLambda([&](FHttpRequestPtr Req, FHttpResponsePtr Res, bool bSuccess){
    OutResponse = Res->GetContentAsString(); // 危险：引用可能失效
    bDone = true;
});
Request->ProcessRequest();
while(!bDone){ FPlatformProcess::Sleep(0.1f); } // 危险：主线程阻塞轮询
```

## 5. 工业级修复原则

1. 不在 Game Thread 把 HTTP 强行 Blocking。
2. 改为纯异步接口（Delegate/Callback/Future），由完成回调推进 Planner。
3. 回调状态使用 `TSharedRef/TSharedPtr` 持有，按值捕获，不捕获栈引用。
4. Owner 销毁时取消请求并解绑回调，避免“幽灵回调”。
5. 超时由 HTTP/Timer 驱动，不做 `while+Sleep`。
6. 所有失败统一映射错误码（`timeout/http_error/invalid_json/contract_invalid`），再走 fallback。

## 6. 安全实现骨架（推荐）

```cpp
struct FReqState
{
    FString Response;
    bool bFinished = false;
};
TSharedRef<FReqState, ESPMode::ThreadSafe> State = MakeShared<FReqState, ESPMode::ThreadSafe>();

Request->OnProcessRequestComplete().BindLambda(
    [State, OnDone](FHttpRequestPtr Req, FHttpResponsePtr Res, bool bOk)
    {
        State->Response = (bOk && Res.IsValid()) ? Res->GetContentAsString() : FString();
        State->bFinished = true;
        OnDone(/*error_code/result*/);
    });
Request->ProcessRequest();
```

## 6.4 二次隐蔽问题与修复（dispatched 后无后续日志）

- 现象：
  - 控制台仅输出 `dispatched ... 异步执行中`，后续 `summary/build_failed` 长时间不出现。
- 根因：
  - 异步状态对象仅通过 `WeakPtr` 传递，缺少强引用保活；
  - 请求发出后状态对象提前释放，回调 `Pin()` 失败直接返回，导致“看起来像超时/卡住”。
- 修复：
  - Runtime 增加“在飞请求强引用池”（in-flight strong refs）；
  - 发起请求时注册状态，完成/失败统一释放状态；
  - 防止回调阶段对象已销毁导致日志链断裂。

## 6.6 为什么仍然触发 E4301（本次最终根因）

- 触发条件（代码默认值）：
  - `enable_thinking=true`
  - `LlmHttpTimeoutMs=12000`
  - `LlmRetryCount=1`（总计 2 次尝试）
- 现象对应：
  - 每次 12 秒到点超时，重试 1 次后 fallback；
  - 日志表现为 `fallback_reason=llm_timeout error_code=E4301 llm_attempts=2`。
- 终端实测（同模型同接口）：
  - `thinking=true, timeout=12s` -> `ERR timeout`（约 `12069ms`）
  - `thinking=false, timeout=12s` -> `OK`（约 `4599ms`）
  - `thinking=false, timeout=30s` -> `OK`（约 `3810ms`）
- 结论：
  - 不是 API 不通，也不是回调丢失；
  - 是“开启思考 + 超时阈值过紧”导致真实请求未在 12 秒内完成。
- 已落地修复：
  - `RealLLMDemo` 默认改为 `enable_thinking=false`、`stream=false`、`timeout=30000ms`。
  - `PlannerBuildOptions` 默认 `bLlmEnableThinking=false`，避免新调用点踩同坑。

## 6.7 新问题：LLM 返回 `args.directory` 导致 `E4009`（已修复）

- 现象：
  - `Probe` 成功（HTTP 200 + raw_response 正常），但 `RealLLMDemo` 报红：
  - `reason=plan_validation_failed code=E4009 field=steps[0].args.directory`。
- 根因：
  - LLM 返回了未声明字段（`directory`），而一期 `args_schema` 为冻结白名单；
  - 校验器对未声明参数执行 Fail-Fast（`arg_not_declared_in_schema` -> `E4009`）。
- 修复策略（不放宽 schema）：
  - 在 `LLM JSON -> Plan` 转换层新增“按 ToolSchema 归一化”：
  - 剔除未声明字段；
  - 对已支持意图工具（`Normalize/LevelRisk/Texture/LOD/ScanAssets`）补齐缺失必填默认值。
  - 在 Runtime 侧新增“返回前强制 ValidatePlan”：
  - 一旦校验失败，统一收敛为 `E4303/llm_contract_invalid`，自动回退 keyword，避免命令端爆红中断。
- 结果：
  - `args_schema` 冻结边界保持不变；
  - Planner 输出稳定（成功则为合法 Plan，失败则自动 fallback，不再出现命令端裸 `E4009` 红错）。

## 7. 回归清单

- 成功请求：结果可用且无阻塞。
- 超时：命中 `timeout` 错误码，无崩溃。
- HTTP 失败：命中 `http_error`，无崩溃。
- Owner 销毁中断：无回调写野指针。
- 高并发多请求：无互相污染，无主线程卡死。
- `Probe` 命令可稳定打印 `raw_response`（用于链路分层排障）。
